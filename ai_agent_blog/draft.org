#+TITLE: unsloth で Gemma-4 を LoRA チューニングした話
#+DATE: 2026-07-19

* きっかけ・概要
- unsloth フレームワークで gemma-4-12B-it-qat-q4_0-unquantized を LoRA チューニングした話
- お題は「ロールプレイを LLM にやらせる」こと。実用目的でのチューニング事例として書く
- Python 環境は uv で構築。pyproject.toml / uv.lock だけで再現できる手軽さに触れる
- RTX 5070 Ti 16GB という限られた VRAM でどう戦ったか（4bit量子化、バッチサイズ1 + 勾配累積8 など）
- 当初考えていたよりはトレーニング時間がかからなかった
- トレーニング時間はかからなかった一方で、環境構築(Gemma-4-12B 用の unsloth)にめちゃくちゃ苦労した


* 環境構築
- python 環境は uv で簡単に構築できるが Gemma-4-12B 用の unsloth が現状バグっている。
- 具体的には以下の URL を参照してもらいたい。
  - https://github.com/unslothai/unsloth/issues/6028
- 簡単にまとめると、
  Gemma-4-12B をファインチューニングするには gemma4_unified 形式に対応する必要があるが、
  gemma4_unified への対応は transformers が 5.10.1 以降の対応が必要。
  一方で unsloth は transformers<=5.5.0 を指定しているため、矛盾が生じている。
- これを回避するため pyproject.toml に以下を指定している
```toml
override-dependencies = [
    "transformers>=5.14.1",
]
```
  

* データセット設計で悩んだこと
- ShareGPT 形式（system + conversations の human/gpt ペア）を採用した理由
- 「役ごとに from を分ける」のではなく「human/gpt のペアが本質」という気づき
  - 学習テンプレートが human → gpt の厳密な交互ターンを前提にしている制約について
- 複数キャラクターを1体のモデルに演じ分けさせる設計
  - human を「ユーザーの発言」ではなく「場面を指示するディレクター役」に変換する発想
  - gpt 側の1ターンに複数キャラの発言を改行区切りでまとめて書かせる方法
- 1ターン1キャラだけ喋らせたい場合の別パターン
  - 「次は◯◯に話させて」という指示 → 1キャラ分の返答、を繰り返す構造
  - 推論時も同じ指示形式を踏襲すれば一貫して使える
- 会話の全履歴を conversations に積むか、要約して system に入れて最後のペアだけ残すか、というトレードオフ
- gpt 側を連続させてよいか（human/gpt 交互という制約とどう付き合うか）
- 1サンプルあたりの往復数の目安（長すぎる conversations の弊害）

* LoRA の汎化性についての疑問
- LoRA で学習した内容は一言一句暗記されるのか、それとも「大まかな流れ」を学習して柔軟に引き出せるのか
- 過学習していない前提での期待値：system プロンプト／ユーザー入力／LLM 応答という構造自体を学習している可能性
- この疑問を軸に、実際に学習してみてどうだったか（今後の検証ネタ）

* スクリプトの作り込み（使い勝手の改善）
- MODEL_NAME をハードコードから --model-name のコマンドラインオプションに変更した経緯
- DATASET_PATH も同様にコマンドラインオプション化
- JSONL をファイル経由だけでなく、メモリ経由（文字列 / Python オブジェクト）でも渡せるようにした設計
  - train.py を「CLI ツール」から「呼び出し可能な関数」として使えるようにする狙い
- データセットが GPG 暗号化されていても復号して読み込めるようにした話
  - plain / 暗号化データの自動判定（ASCII armor / バイナリ OpenPGP の検出）
  - 対称鍵（パスフレーズ）・公開鍵（gpg-agent 経由）の両対応
  - ロールプレイ用データセットのような扱いに気を遣いたいデータをどう安全に管理するか、という運用観点

* トレーニング時間

- 今回始めて LLM の LoRA チューニングを実施してみた
- ResNet や CLIP による画像識別モデルのチューニングはそこそこやっているが、
  それらは結構時間がかかる、かつ、ロスがなかなか下らなかったりする。
- また、やったことはないが画像生成系の LoRA は結構時間がかかると聞いていたので、
  今回の Gemma-4-12B もめちゃめちゃ時間がかかるかもと思っていた。
- しかし、実際にやってみると、個人が用意できるような規模のデータセットなら
  1 時間もかからずに LoRA のチューニングが可能。
- VRAM も 16GB で十分余裕がある。 (8GB ではキツい)
  これは unsloth の恩恵だろう。
- LoRA トレーニングのデータセットの用意は面倒だが、
  トレーニングの結果はそれに見合うだけの効果は得られる、
  というのが今回の経験から得られた知見。
- 例えば、参考書をスキャンして PDF 化 & OCR してデータセットを作成してトレーニングすれば、
  マイナーな資格試験に特化した個人用 LLM も簡単に作れそうだと感じた。
- なお、作成した LoRA の権利面はトレーニングデータによって色々とあると思うので、
  公開する場合は注意が必要。
