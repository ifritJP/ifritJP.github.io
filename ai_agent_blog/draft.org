- 2026/6/1 から GitHub Copilot のクレジットの扱いが大きく変更された。
- この変更により、バイブコーディングでアプリ開発すると、小規模なアプリでも1日でクレジットを使いきってしまうことが判った。
- というか、クレジットを使い切ってアプリが完成するならまだしも、アプリは完成していない。
- そんな訳で、クラウドサービスに頼らないローカル LLM でどこまでバイブコーディングが可能か？
  を調べてみた。
- 具体的には「VRAM 16GB で動くローカル LLM でバイブコーディングするならどのモデルが一番マシか」

- 実験準備
  - 今回のバイブコーディングのお題は、「端末上(CUI)で動く簡易テトリス」の開発
  - 競技プログラミング的な入出力処理よりも、この方がインタラクティブ処理が必要で更に難易度が高いだろうか？
  - 今回試した AI モデルは以下。
    - Gemma4-12B
    - QWen3.6-27B-Q3
    - QWen3.5-27B-Q4
    - QWen3.5-27B-Q3
    - QWen3.5-9B-Q8
    - QWen2.5-Coder-14B
  - また、 AIエージェントは次のものを試した。
    - aider
    - opencode
    - cline
  - なお、 AI モデルと AI エージェントは全組合わせを確認した訳ではない
  - 特に AI エージェントは、ほぼ上から順に試していって、結果的にほぼ cline しか使っていない
  - 今回の目的である「バイブコーディング」には、上記2つはあまり向かないようだった

- 実験方法
  - そこそこまともなバイブコーディング環境を目指すので、
    AI の重みと KV キャッシュは GPU に全て載せられることを前提にする。
  - そのため、モデルによってはコンテキストウィンドウサイズがかなり制限される
  - プロンプトは最小限のフワっとした感じの情報だけ与える
  - 細かい仕様は、出力結果を見ながら追加で与える
  - 開発言語は python とする
  - python は uv でセットアップ済みの状態    

- 実験結果
  - 今回のお題をまともにクリアできたのは *QWen3.6-27B-Q3, QWen3.5-27B-Q3* だけだった。
  - 「まとも」と言っても、テトリスのこまかいルールまで実装させていない。
  - 実装したのは、「テトリミノを動かし、1行揃ったら消す」というテトリスの超基本ルールまで。
  - プログラミングを自主的に勉強した人なら、誰でも一度は作ったことがあるんじゃないか？
    という極々基本的な処理といえます。
  - しかし、こんな基本的な処理ですら、他のモデルではさっぱりと完成させられませんでした。
  - 何度もプロンプトで指示を出しても改善する見通しがなかったです。

  - なお、クリアできた 2 つのモデルも、ストレスなく出来た訳ではなく、かなりの時間待たされました。
  - また、今回は対象が 1 ファイルの非常に小規模なコードベースでした。
  - それでもかなりの時間待たされたので、実務で利用するコードベースではまず現実的とは言えないでしょう。

  - Gemma4-12B で出来なかったのは意外でした。
  - また、今回お題をクリアした QWen3.6-27B, QWen3.5-27B は、
    VRAM 16GB の制約から Q3 モデルを使っていたのでかなり精度は落ちているはず。
    それでも他のモデルに比べても、結果としてはレベルが高い結果になったのはちょっと驚きでした。
  - やはり *LLM はパラメータのデカさがなによりも性能に影響する* ということでしょう。
  - そうなると、 ローカル LLM において dGPU の *VRAM のサイズ制約は非常に厳しい* です。
  - 現状の Apple の Mac や NVidia の DGX Spark, RTX Spark は、
     CPU(GPU), RAM 間のインタフェースが直結なので、サイズ、帯域問題をクリアしているが、
    これだとメモリを後から拡張するのが実質的に不可能になる。
  - 将来 システム RAM が DDR6 になったとしても、
    dGPU とのインタフェースが改善されて帯域が確保されない限り、
    一般的な PC でコスパの良いローカル LLM 処理を実現するのは難しい。
  - この辺りのブレークスルーに期待したい


  - なお、以下のサイトは各 AI モデルの様々な性能を比較確認できる。
    - https://artificialanalysis.ai/models/claude-opus-4-8?intelligence=coding-index
  - このサイトを見ると、Coding の能力を測るスコアについて以下のようなことが分かる。
    - Qwen3.6 27B は gpt-oss-120B よりも性能が高い
    - Qwen3.6 27B のスコアは 36.5、Qwen3.5 27B のスコアは 34.9、Qwen3.5 9B のスコアは 25.3

  - このスコアと今回の実験結果をまとめると、コーディングエージェント利用するには、
    このスコアが少なくとも 30 以上は必要だということが言えそう。

  - そして、 スコア 30 以上というと、
    Claude 3.5 Sonnet, Claude 4 Sonnet, GPT-5, Gemini2.5 Pro など、
    ちょっと前まで現役でコーディングエージェントとして使われていた AI が挙がっているので、
    概ね認識としては間違っていないだろう。

  - つまり、QWen3.6-27B をもっと快適に使える環境さえあれば、クラウドサービスに頼らずに、
    ローカルでバイブコーディングを実行することが可能になる、と言えるだろう。

  - そのための現状最も現実的な解は、RTX5090 となる。

  - いやいや、高すぎだから。。。

  - LLM に限定すれば、もしかしたら AMD Radeon AI PRO R9700 も選択肢になるのかもしれない。
    値段も消費電力も約 RTX5090 と比べると約半分。ただし、性能は 1/3。
    LLM 以外の AI 利用には、まだまだハードルが高いけど、
     LLM に限定すれば、ほぼ問題なく動くレベルなので。
    知らんけど。
